# Feature parity: trading-backtester vs известные системы бэктестинга

Дата анализа: 2026-07-12.
Метод: deep-research по внешним источникам (5 поисковых направлений, ~30 выгруженных
источников, частичная адверсариальная верификация) + инвентаризация нашего кода и ROADMAP.
Статус верификации утверждений — в разделе «Надёжность источников» внизу.

## TL;DR

Наши уникальные сильные стороны — детерминированный `result_hash` с golden-гейтами,
Ed25519-подписанные evidence-бандлы, Docker-песочница для недоверенного LLM-кода и
machine-readable authoring doc — не имеет **ни один** из рассмотренных конкурентов
(включая LEAN, NautilusTrader, VectorBT PRO). Свежий обзор LLM-трейдинг-агентов
(arXiv 2605.19337) показывает: из 19 эмпирических работ только 2 имеют честные
train/test-разрезы и 0 достигают высшего уровня воспроизводимости — мы уже выше всей
области по этой оси.

Главный разрыв — **статистическая строгость против оверфиттинга**. У нас robustness-проверки
объявлены в контракте, но не вычисляются (`DeferredRobustness: 'validated_but_not_computed'`),
метрик всего 7, оптимизация — только grid. Именно этот слой решает, будет ли
LLM-цикл «гипотеза → бэктест → уточнение» производить допустимые к живой торговле стратегии
или итеративно подгонять их под тестовый период. Топ-5 предлагаемых фич: (1) учёт числа
испытаний + Deflated Sharpe gate, (2) walk-forward/CPCV-разрезы как режим прогона,
(3) серверный held-out OOS-период, скрытый от lab, (4) корреляционный novelty-гейт гипотез
против пула принятых стратегий, (5) структурированный failure-feedback для LLM.

## 1. Наша система: что уже есть (ground truth)

Сервисный детерминированный research-бэктестер (TypeScript):

- HTTP API + async-lifecycle задач, идемпотентность, cancel, webhook, backpressure (429),
  Pg-очередь с горизонтальными воркерами (SKIP LOCKED, lease/heartbeat), dedup по
  fingerprint + in-flight coalescing, LISTEN/NOTIFY, bundle-by-ref.
- **Детерминизм как инвариант**: байт-идентичный `result_hash` при повторе, golden-гейты
  на каждом перф-механизме, content-addressed артефакты и бандлы.
- **Песочница для недоверенных LLM-бандлов**: Docker без сети, cap-drop, лимиты ресурсов,
  чистая таксономия ошибок, per-bar IPC, universe-сессии (один контейнер на бандл,
  N инстансов по символам).
- **Baseline-vs-variant overlay-движок** с дельтами метрик; registry discovery + пресеты.
- **SDK с authoring-поверхностью для LLM**: machine-readable authoring doc, документирующий
  runtime API `ctx.market` (OHLCV + open interest + ликвидации + funding + taker flow,
  as-of/window-запросы, point-in-time-корректность).
- **Ed25519-подписанные evidence-бандлы** для допуска стратегий на живую платформу;
  сверка execution/PnL с живым paper-движком (остаток 3e-7, модель `same_bar_close`);
  trade-reconcile-харнесс; opt-in funding-модель реализма.
- Lab-сторона: LLM-генерация гипотез, `ParamGridRunner` с ограниченным параллелизмом.

Подтверждённые пробелы (по коду):

- Метрики: только `pnl`, `max_drawdown`, `win_rate`, `sharpe`, `total_trades`,
  `profit_factor`, `top_trade_contribution_pct` (`engine/metrics.ts::computeMetrics`).
- Robustness-проверки: интерфейс `DeferredRobustness` со статусом
  `'validated_but_not_computed'` — валидируются, но не считаются.
- Оптимизация — только grid; нет walk-forward/CV; нет tear-sheet; нет slippage/latency-моделей;
  один таймфрейм на прогон; нет портфельного уровня; novelty гипотез — только точный fingerprint.

### 2.2. Платформы альфа-майнинга (admission-механики — самое релевантное нам)

**WorldQuant BRAIN:**

- Альфа допускается к скорингу **только после квалификации out-of-sample** — IS-результат
  сам по себе ничего не значит.
- Корреляционный dedup против пула: корреляция **дневных PnL-дельт за 2 года**;
  порог с escape-hatch — коррелирующая альфа допускается, если Sharpe лучше на ≥10 %.
- Гипотезы пишутся на ограниченном expression-DSL с фиксированными операторами — это
  одновременно и безопасность (наш аналог — песочница), и сужение пространства поиска.

**Numerai:**

- **MMC**: сабмишен ортогонализуется против stake-weighted мета-модели, оплачивается только
  ортогональный остаток — дублирующий сигнал получает ноль. Считается простой линейной
  алгеброй по векторам предсказаний.
- **TC**: вклад = градиент пост-оптимизационной доходности портфеля по ставке на сигнал
  (cvxpylayers, дифференцируемый оптимизатор в PyTorch) — оценивается маржинальная
  полезность в портфеле, а не standalone-корреляция.

### 2.3. LLM-системы генерации стратегий (2023–2026)

**AlphaAgent (KDD 2025, arXiv 2502.16789)** — самый близкий к нашей архитектуре паттерн:

- Замкнутый цикл Idea Agent → Factor Agent → Eval Agent; фидбек бэктеста двигает следующий
  раунд уточнения гипотезы.
- Novelty через **AST-подобие** (наибольшее общее поддерево) против «зоопарка» известных
  альф — слишком похожие штрафуются.
- **Регуляризатор при генерации**: штрафы за символьную длину, число параметров,
  избыточные фичи — оверфиттинг подавляется до бэктеста.
- LLM-судья проверяет согласованность «гипотеза ↔ описание фактора ↔ код».
- Machine-readable фидбек: IC/RankIC, годовая доходность, IR, MDD + **категоризация причин
  отказа**, накапливаемая в базе знаний агента.
- KPI цикла: hit ratio (доля альф выше порога за раунд) и development success rate (доля
  исполнившихся без ошибок). С включёнными ограничениями hit ratio 0.16 → 0.29 (+81 %).

**AlphaMemo (arXiv 2606.20625)** — память о процессе поиска:

- Запоминаются не финальные скоры, а тройки «контекст родительского фактора → AST-diff
  правки → исход» со статусом invalid/rejected/admitted.
- **Асимметричное вето**: негативные паттерны (невалидные выражения, перемайненные
  трансформации, высококоррелирующие правки) жёстко блокируют кандидатов до генерации;
  позитивные — только мягкий буст. Обоснование: негативные свидетельства стабильнее
  по режимам рынка, чем хрупкие позитивные.
- Числовые гейты допуска: |ICIR| ≥ 0.10; максимальная |корреляция Пирсона| с пулом ≤ 0.70;
  ёмкость пула 50; порог сложности (длина фактора) 40.
- Строгая IS/OOS-дисциплина: память обновляется только с train/validation, тест — только
  для финального отчёта.

**Обзор «Agentic Trading» (arXiv 2605.19337)** — аудит строгости 19 работ:

- 2/19 — извлекаемые time-consistent разрезы; 1/19 — явная модель транзакционных издержек;
  1/19 — survivorship; 15/19 — низший уровень воспроизводимости R0; 0/19 — R3.
- Главный риск валидности: **adaptive overfitting из самого LLM-цикла** — итеративное
  уточнение против валидационных метрик надувает результат, если харнесс не навязывает
  single-use тест или walk-forward.
- Рекомендуемый минимум: time-consistent разрезы, явные издержки, timestamp-дисциплина,
  раскрытие бюджета поиска (число попыток), Deflated Sharpe / Reality Check / SPA.
- **Outcome Embargo** для памяти агента: эпизод, записанный в момент t, не должен отдавать
  своё поле «исход» в retrieval до t+k — канал утечки, специфичный именно для LLM-агентов.

**Interpretable Hypothesis-Driven Trading (arXiv 2512.12924)**: walk-forward на 34
независимых OOS-периодах; честный результат под строгой валидацией — Sharpe 0.33,
p=0.34 (незначимо) — против типичных публикуемых 15–30 % годовых; иллюстрация того,
что строгий харнесс отсекает большинство «успешных» стратегий.

**Прочие**: Chain-of-Alpha (фидбек: IC, turnover, decay), AlphaLogics (библиотека
«рыночных логик» как организация пространства гипотез), QuantAgent (writer/judge),
Alpha-GPT (human-in-the-loop), FinAgent (слоистая память + рефлексия; point-in-time
контролируется только промптом — риск утечки).

### 2.4. Статистический инструментарий против оверфиттинга

- **Deflated Sharpe Ratio** (Bailey & López de Prado): корректирует Sharpe на selection bias
  при множественном тестировании. Ключевое: бэктест без раскрытого числа испытаний N —
  «worthless»; holdout не спасает при многократном применении. Рабочий пример: годовой
  SR 2.5 за 5 лет **проваливает** 95 %-DSR-гейт при N=100 попыток, но проходит при N=46 —
  порог допуска обязан масштабироваться с числом записанных попыток.
- **Arian et al. 2024** (Knowledge-Based Systems): на синтетике с известной ground truth
  **CPCV заметно превосходит** K-Fold, Purged K-Fold и Walk-Forward по PBO и DSR;
  walk-forward — слабейший в предотвращении ложных открытий; чистый purging без
  комбинаторики почти ничего не добавляет.
- Реализации: VectorBT PRO (purged WF + combinatorial CV, с релиза v2024.4.1), skfolio
  (`WalkForward`, `CombinatorialPurgedCV`, purging + embargo), PyBroker (WF + bootstrap
  доверительных интервалов метрик). mlfinlab ушёл в платный closed-source — ориентир
  по спецификации, не зависимость.

## 4. Предлагаемые фичи (ранжировано)

### Tier 1 — анти-оверфиттинг для LLM-цикла (решает судьбу продукта)

**1. Реестр испытаний + Deflated Sharpe gate.**
Сервер уже видит каждый прогон каждой гипотезы (fingerprint, dedup, lab-сабмиты) — мы в
уникальной позиции вести честный счётчик N испытаний на семейство гипотез автоматически,
чего не делает никто из OSS. Дальше: DSR как вычисляемая метрика, порог допуска
масштабируется с N; N и DSR включаются в подписанный evidence-бандл (продолжение нашей
сильной стороны — «доказуемая строгость», а не только «доказуемая воспроизводимость»).
Это прямая реализация рекомендаций обзора Agentic Trading и Bailey/Лопеса де Прадо.
Стоимость: низкая (учёт — расширение существующего fingerprint-слоя; DSR — формула
над уже считаемым Sharpe + моменты распределения).

**2. Walk-forward и CPCV-разрезы как first-class режим прогона.**
Новый параметр запроса: схема разреза (rolling/expanding walk-forward; позже combinatorial
purged CV с embargo). Движок детерминирован — разрезы дают детерминированный набор
под-прогонов, агрегируемых в один результат с per-fold метриками и PBO. Дизайн-ориентиры:
`Splitter` из VectorBT PRO, `CombinatorialPurgedCV` из skfolio, выводы Arian et al.
(walk-forward — минимум, CPCV — цель). Дедуп/коалесцинг фолдов у нас уже бесплатно
работает (каждый фолд — отдельный computeIdentity).
Стоимость: средняя; полностью ложится на существующую архитектуру очереди.

**3. Серверный held-out OOS-период («квалификация» как в BRAIN).**
Часть истории объявляется квалификационной: lab (и LLM) не может запускать против неё
произвольные итерации — только финальную заявку на допуск, single-use или с жёстким
бюджетом. Это единственная системная защита от adaptive overfitting самого LLM-цикла
(итерации против видимых метрик). Двойник на lab-стороне — Outcome Embargo для памяти
агента (исход эпизода недоступен retrieval до t+k).
Стоимость: средняя (политика поверх существующего датасет-слоя + запись в evidence).

**4. Novelty-гейт гипотез против пула принятых стратегий.**
Три уровня, по нарастающей: (а) корреляция дневных PnL-дельт кандидата с каждым членом
пула (BRAIN: окно 2 года, escape-hatch при Sharpe +10 %; AlphaMemo: |ρ| ≤ 0.70) — у нас
уже есть все траектории equity в артефактах; (б) AST-подобие бандлов (наибольшее общее
поддерево, AlphaAgent) — бандлы JS/TS парсятся стандартно; (в) позже — MMC-style
ортогонализация против «мета-портфеля» принятых стратегий. Результат — числовой
novelty-score в `RunResultSummary`, который lab использует как сигнал в цикле уточнения.
Стоимость: (а) низкая, (б) средняя, (в) исследовательская.

**5. Структурированный failure-feedback для LLM.**
Расширить результат машиночитаемым разбором: вектор качества
{Sharpe, Sortino, MDD, turnover, decay, IC/ICIR — где применимо}, категория отказа
(нет входов / оверфит-подозрение / нарушение сложности / расхождение с гипотезой),
per-trade диагностика как артефакт. Плюс KPI цикла на lab-стороне: hit ratio и
development success rate по раундам (AlphaAgent показал +81 % hit ratio от такой
дисциплины). Наш authoring doc — уже половина этого моста; это вторая половина
(«почему не сработало» тем же machine-readable каналом, что и «как писать»).
Стоимость: низкая-средняя, чисто аддитивная.

### Tier 2 — реализм и наблюдаемость

**6. Slippage/fill-модели с seeded-стохастикой.** Полка конфигурируемых моделей по образцу
NautilusTrader (fixed-tick, volume-sensitive, probabilistic с `random_seed`) — наш
детерминизм сохраняется (seed уже в контракте запроса). Практики отмечают 20–50 %
живых haircut'ов даже при «валидированных» точечных fill-моделях — сейчас LLM-стратегии
оптимизируются против нереалистично добрых исполнений.

**7. Расширенный каталог метрик.** Sortino, Calmar, CAGR, expectancy, SQN, rolling Sharpe,
разбивки по символам/причинам выхода/периодам (уровень Freqtrade). Дёшево: чистые функции
над уже существующими equity/trades.

**8. HTML-tearsheet как вид артефакта.** Паттерн Nautilus `create_tearsheet_from_stats` —
отчёт строится из готовой статистики без живого движка, что идеально для нашей сервисной
архитектуры: self-contained HTML кладётся в content-addressed artifact store. Закрывает
и «streamed progress» частично (ROADMAP item 19 — смежный).

**9. Оптимизация за пределами grid (lab-сторона).** Optuna в `ParamGridRunner`
(TPE/NSGA-III), типизированные параметр-пространства в манифесте SDK (Int/Decimal/
Categorical с дефолтной точностью 3 знака — анти-оверфит-дефолт Freqtrade), ленивый
random search по большим сеткам, отчёт «поверхностей» (устойчивые регионы вместо
точечного оптимума — VBT PRO).

### Tier 3 — по мере роста продукта

**10. `timeframe-detail`** — интра-свечная переоценка активных свечей на младшем ТФ
(Freqtrade) и/или 4-точечный OHLC-путь бара (Nautilus).
**11. Портфельный уровень** — общий капитал на несколько стратегий/символов, exposure caps
(VBT PRO group-семантика). Большая работа; сейчас product-решение «последовательный
проход по портфелю» зафиксировано сознательно.
**12. Память процесса поиска (AlphaMemo) на lab-стороне** — журнал AST-diff правок
с исходами + асимметричное вето; строится поверх фич 4–5.
**13. SandboxEval-style containment-тесты** — регрессионный набор на побег/exfiltration
для нашей песочницы; усиливает историю к gVisor-апгрейду (ROADMAP item 18).
**14. LLM-слой над документацией движка** (ChatVBT-паттерн) — семантический поиск по
authoring doc/SDK для агентов; у нас уже есть machine-readable основа.

### Чего сознательно НЕ делать

- Не гнаться за L2/L3-матчингом Nautilus — наш продукт research-гейтинг гипотез на барах,
  а не HFT-симуляция; выигрыш по допуску стратегий даёт Tier 1, а не глубина стакана.
- Не строить свою облачную data-платформу уровня LEAN — данные приходят через
  platform-контракт, это уже решено архитектурно.
- Не ослаблять детерминизм ради стохастического реализма — только seeded-модели
  (byte-identical при том же seed), golden-гейты остаются merge-баром.

## 5. Что даст внедрение топ-5 и как это ложится на ROADMAP

Синхронизировано с `docs/ROADMAP.md` — добавлена **Phase E (items 20–24)**. Ниже — обоснование
ценности и порядка.

### 5.1. Ценность каждого предложения

**E1 (метрики + структурированный failure-feedback) — субстрат и скорость цикла.**
Прямой эффект — конверсия LLM-цикла: AlphaAgent поднял hit ratio раундов с 0.16 до 0.29 (+81 %)
именно дисциплиной «машиночитаемая причина отказа → база знаний → следующий раунд». В наших
терминах: каждый лишний прогон — это ~23 с движка на воркер-слот; меньше пустых итераций на одну
принятую стратегию = прямая экономия compute и LLM-токенов lab. Побочный эффект — расширенные
метрики нужны всем остальным пунктам (DSR требует моменты распределения, WF — per-fold метрики,
novelty — траектории PnL). Важно: новые метрики аддитивны через существующий механизм
запрошенных `metrics` — незапрошенные не считаются, старые запросы остаются байт-идентичными.

**E2 (реестр испытаний + DSR-гейт) — защита от главного системного риска.**
Вся наша работа Phase C/D по пропускной способности (dedup, coalescing, горизонтальные воркеры,
universe-сессии) удешевляет перебор вариантов — и тем самым **усиливает** selection bias: из
сотен вариантов что-то покажет Sharpe 2+ чисто случайно. Bailey/Лопес де Прадо: SR 2.5 за 5 лет
проваливает 95 %-гейт уже при N=100 попыток. Мы — единственные, кто может вести счётчик N честно
и автоматически (сервер видит каждый прогон через fingerprint; у OSS-движков перебор происходит
на стороне пользователя и N никому не известен). N + DSR в подписанном evidence превращает наш
козырь «доказуемая воспроизводимость» в «доказуемая строгость» — этого нет ни у кого. Запуск в
два шага: сначала advisory-метрика в отчёте, потом гейт.

**E3 (walk-forward → CPCV) — устойчивость вместо подгонки под окно.**
Сегодня один прогон = одно окно истории; стратегия, случайно подходящая к 2024 году, неотличима
от рабочей. WF даёт per-fold стабильность (паттерн arXiv 2512.12924: 34 независимых OOS-периода
отсекают большинство «успешных» стратегий), CPCV добавляет PBO — вероятность, что бэктест
оверфитнут. Инфраструктурная синергия: фолды — обычные джобы с собственным computeIdentity,
значит дедуп, коалесцинг и горизонтальные воркеры работают per-fold бесплатно, а WF-прогоны —
это ровно та нагрузка, которая потом заново обоснует размороженный перф-трек (Tier 3).

**E4 (held-out OOS-квалификация) — закрывает то, что E2/E3 не закрывают.**
Обзор Agentic Trading называет главным риском валидности adaptive overfitting самого цикла:
LLM итерирует против видимых метрик, и информация о тест-периоде утекает в промпт-цикл — даже
при честном WF и DSR. Единственная системная защита — период, против которого итерации
невозможны: финальная заявка single-use (модель квалификации WorldQuant BRAIN: скоринг только
после OOS-квалификации). Продуктовый смысл: штамп «qualified OOS» в evidence делает допуск на
живую платформу осмысленным, а не самосертификацией lab.

**E5 (novelty-гейт) — разнообразие пула вместо 50 перефразировок одного momentum.**
Без него LLM-фабрика сходится к переоткрытию одного и того же шаблона (AlphaAgent: гомогенные
факторы → crowding → ускоренный alpha decay). Корреляция PnL-дельт против пула принятых
стратегий — дешёвый пост-процессинг над equity-артефактами, которые мы уже храним. Экономика
доказана BRAIN/Numerai: дублирующий сигнал стоит ноль (MMC платит только за ортогональный
остаток). Novelty-score в `RunResultSummary` — это ещё и reward-сигнал для цикла: LLM начинает
исследовать, а не эксплуатировать известный шаблон.

**Суммарно** воронка меняется с «генерация → бэктест на одном окне → допуск» на:
генерация (novelty-фильтр) → WF/CPCV-прогон → структурированный фидбек → квалификация на
held-out OOS → DSR-гейт по числу попыток → evidence с {N, DSR, PBO, novelty, OOS-вердикт}.
Продукт из «бэктестера» становится **машиной допуска, чьим штампам можно верить** — а это и есть
ценностное предложение всей связки lab → backtester → platform.

### 5.2. Последовательность и синхронизация с ROADMAP

| Порядок | Item | Размер | Почему на этом месте |
|---|---|---|---|
| 1 | E1 (20) метрики + фидбек | S | Субстрат для E2–E5; мгновенная польза lab-циклу; чисто аддитивно |
| 2 | E2 (21) реестр N + DSR | S–M | Дёшево (расширение fingerprint-слоя), требует моменты из E1; advisory → гейт |
| 3 | E3 (22) walk-forward | M | Требует per-fold метрики E1; ложится на очередь как есть; CPCV — follow-up |
| 4 | E4 (23) held-out OOS | M | Строится на split-механике E3 + policy-решение (окно, бюджет) |
| 5 | E5 (24) novelty | S–M | Параллелизуем с E3/E4 после E1; сначала PnL-корреляция, AST — потом |

Взаимодействие с существующими треками:

- **Перф-лестница остаётся на паузе** (решение зафиксировано): bar-major Slice B, 17b-тюнинг,
  Tier 3 scale-out не блокируют Phase E и не нужны ей. Наоборот — WF/CPCV породит кратный рост
  числа джобов, который и станет тем «реальным бутылочным горлышком», после которого перф-трек
  размораживается по правилам item 17.
- **Item 19 (streamed progress)** и **item 18 (B2C-гейт)** — ортогональны, приоритет не меняется.
- **Tier 2 из §4** (slippage-модели, tearsheet, Optuna в lab) — после E1–E5. Отдельно про
  slippage: для допуска на НАШУ платформу fill-модель уже честна по построению (сверка с живым
  paper-движком до 3e-7), поэтому slippage — не про допуск, а про будущий выход за пределы
  paper-консистентности.
- Каждый слайс держит golden-gate merge-бар: детерминизм и байт-идентичность OFF-пути — инвариант.

### 5.3. Идентичность семейства гипотез (решение 2026-07-12)

Счётчик испытаний N (E2) осмыслен, только если правильно определено «семейство» — какие прогоны
считать попытками одной идеи. Узкое определение обходится переименованием гипотезы или
переписыванием бандла (счётчик обнуляется — гейт декоративный); слишком широкое наказывает
честные новые идеи чужими попытками. Решение — четыре слоя, от дешёвого к авторитетному:

| Слой | Механизм | Что ловит | Когда | Статус |
|---|---|---|---|---|
| L0 | fingerprint | Точный дубль запроса | До прогона | Есть |
| L1 | hypothesis id (lab) + lineage `derivedFrom` в манифесте бандла | Явное родство: правки кода, та же гипотеза | До прогона | `derivedFrom` — новое поле, малая правка контракта |
| L2 | Семантический поиск похожих гипотез на стороне lab: `SimilarHypothesisSearchPort` (лексический Jaccard) + `PgHybridStrategySimilarityAdapter` (FTS/RRF) | Перефразировки до кодогенерации и сабмита | **До прогона** — единственный слой, экономящий прогон целиком | Порт и адаптеры уже реализованы (advisory); включить в pre-submit + прокидывать `familyHint`; эмбеддинги/pgvector — позже |
| L3 | Корреляция дневных PnL-дельт (E5) | Поведенческое родство независимо от формулировки | После прогона | Часть E5 |

Иерархия авторитетов повторяет принцип, уже закреплённый в lab `EvidencePolicy`: fingerprint —
единственный авторитет «то же самое»; семантика — всегда «похоже», никогда «то же»; финальный
арбитр родства — поведение (L3): для DSR статистически важна именно коррелированность испытаний
(коррелирующие попытки сокращают *эффективный* N). L3 задним числом склеивает семейства,
пропущенные слоями L1–L2, и корректирует novelty-score.

Бонус L2 — RAG-память генерации (паттерн AlphaMemo): retrieval отдаёт LLM исходы похожих гипотез
(«вот три родственные, вот их метрики и категории отказов»), умножая эффект E1. Обязательное
ограничение — **Outcome Embargo**: в контекст генерации нельзя отдавать исходы квалификационного
OOS-периода (E4), иначе RAG-слой сам становится каналом утечки теста.
